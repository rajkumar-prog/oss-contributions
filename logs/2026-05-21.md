## What I worked on

Fixed two issues in two different repos тАФ a network protocol issue and a math bug.

**Fix 1 тАФ Retry-After header in Weave Node SDK (PR #6922):**
When you send too many requests to a server, it responds with HTTP 429 (Too Many Requests). The server can also send a `Retry-After` header saying exactly how long to wait. The Weave Node SDK already had retry logic, but it completely ignored this header тАФ it would wait 100ms and retry, even if the server said "wait 30 seconds." Result: all retries got rejected too, and the request failed for no good reason.

Added a `parseRetryAfterMs()` function that reads the header (which can be in two formats тАФ a number of seconds, or a specific date/time). If the wait is reasonable (under 60 seconds), use it. If the server says wait 2 hours, give up and return the error immediately.

An automated code review (Codex) caught two edge cases: the 429 logic was bypassing the `retryOnStatus` callback, and negative header values weren't being rejected. Both fixed.

**Fix 2 тАФ FactualCorrectness recall bug in ragas (PR #2737):**
This was the most meaningful fix of the week. `FactualCorrectness` is a metric that measures how factually accurate an AI's response is. It works by running NLI (Natural Language Inference) in two directions:
- `reference_response`: checks which response claims are supported by the reference тЖТ precision direction
- `response_reference`: checks which reference claims are covered by the response тЖТ recall direction

The bug: both recall and F1 modes were using `reference_response` (precision TP) instead of `response_reference` (recall TP). So recall scores were silently inflated.

Concrete example: expected recall was 0.667 (2 out of 3 reference claims covered). The old code returned 0.75 (3 out of 4 response claims supported) тАФ a completely different number measuring a completely different thing.

Fix: renamed `tp` to `tp_precision` and added `tp_recall` тАФ making it impossible to mix them up. Added 9 unit tests, including one that explicitly checks the old wrong value (0.75) is no longer returned.
