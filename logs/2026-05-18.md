# 2026-05-18

## What I worked on

Fixed a security vulnerability in the openai/evals Flask server — PR #1669.

Inside openai/evals there's a mini web server that lets an AI agent control a browser (click buttons, fill forms). When something went wrong in this server, the code was doing `return jsonify({"error": f"Something failed: {e}"})` — where `{e}` dumps the full Python exception into the API response. That exception contains file paths, line numbers, library versions — basically a map of the server's internals.

This is called **information disclosure** — accidentally telling the outside world more than they need to know. Attackers can use that information to find known vulnerabilities in specific library versions, or understand the file structure of the server.

The fix: separate the two audiences. The *client* gets a generic "Navigation failed" message. The *developer* gets the full error in server logs via `logger.error(f"Navigation failed: {e}")`. Nothing is lost for debugging. Nothing is leaked to the outside.

Changed 4 locations in the file, all following the same pattern.

**Deep note:** This is one of the most common accidental security mistakes in web APIs. The rule is simple — never put raw exception objects in API responses. Log them internally, return generic messages externally.
